# Как использовать extension в проекте (Qwen CLI)

Эта инструкция описывает типовой сценарий для фронтов:

1. У вас есть спека аналитика (описание методов бэка, бизнес-логика, критерии приёмки).
2. У вас есть дизайн (Pixso) или хотя бы ссылка/скриншоты.
3. Вы хотите реализовать фронт с переиспользованием core UI и проектных компонентов, с тестами и проверками.

## 0) Предпосылки

- У вас установлен `qwen` (Qwen Code CLI).
- У вас установлен наш extension `qoder-frontend-workflow`.
- Вы запускаете Qwen из корня нужного фронтового репозитория.

Проверить, что extension виден:

```bash
qwen extensions list
```

Должно быть что-то вроде:

- `qoder-frontend-workflow (0.1.0)`
- Commands: `/work:orchestrate`, `/work:validate-spec`, `/work:plan`, `/work:review`, `/work:pr-summary`

## 1) Подключение extension

Если extension ещё не установлен:

```bash
qwen extensions install https://github.com/lancelap/qoder-front-ext.git
```

Если хотите обновиться:

```bash
qwen extensions update qoder-frontend-workflow
```

## 2) Подготовка проекта (один раз на репозиторий)

Ключевая идея:

- Общие правила workflow и команды живут в extension.
- Контекст конкретного проекта живёт в репозитории проекта (`QWEN.md` и `.qwen/`).

### 2.1) Добавьте проектный `QWEN.md` и `.qwen/`

В вашем фронтовом репозитории создайте:

- `QWEN.md`
- `.qwen/skills/...`
- `.qwen/agents/...`

Шаблоны лежат в extension:

- `templates/project-qwen/QWEN.md`
- `templates/project-qwen/.qwen/skills/*/SKILL.md`
- `templates/project-qwen/.qwen/agents/domain-expert.md`

Скопируйте их в проект и замените примеры на реальные факты проекта.

### 2.2) Заполните `component-inventory` (это критично для переиспользования)

Заполните файл:

- `.qwen/skills/component-inventory/SKILL.md`

Минимум, который там должен быть:

- Core UI импорты (кнопка, инпут, модалка, таблица, табы, нотификации).
- 5–20 крупных проектных компонентов, которые “уже есть на многих страницах”, с правилами:
  - когда использовать
  - когда не использовать
  - где примеры в коде

Это заменяет отдельный RAG на MVP: агент сначала читает inventory, потом ищет по проекту, и только потом создаёт новый компонент.

### 2.3) Опишите проверки проекта

Заполните файл:

- `.qwen/skills/local-verification/SKILL.md`

Там должны быть реальные команды:

- тесты
- typecheck (если есть)
- lint (если есть)
- build (когда нужно)
- manual visual-check (когда нужно)

## 3) Типовой workflow “есть спека → делаем фронт”

Ниже практичный сценарий, который повторяется для каждой задачи.

### 3.1) Положите спеку и дизайн рядом с кодом (рекомендуется)

В проекте создайте, например:

- `docs/specs/GB-123.md` (спека аналитика)
- `docs/design/GB-123.md` (ссылка на Pixso + краткое описание экранов/состояний)

Почему это удобно:

- Qwen сможет ссылаться на конкретные файлы.
- Спека/дизайн остаются в истории проекта.

### 3.2) Запустите Qwen в корне проекта

Интерактивный режим:

```bash
qwen
```

Или сразу стартануть с промптом и остаться в интерактиве:

```bash
qwen -i "Нужно реализовать задачу GB-123. Спека: docs/specs/GB-123.md. Дизайн: docs/design/GB-123.md."
```

### 3.3) Запустите оркестратор

Внутри Qwen-чата:

```text
/work:orchestrate Реализовать GB-123. Спека: docs/specs/GB-123.md. Дизайн: docs/design/GB-123.md.
```

Что должно произойти по процессу:

1. Валидируется спека (готово/заблокировано + вопросы).
2. Валидируется дизайн (готово/риски/заблокировано + расхождения).
3. `project-analyst` ищет похожие страницы и компоненты для переиспользования.
4. Оркестратор делает план (файлы, шаги, тесты, команды проверок).
5. Оркестратор останавливается и просит подтверждение плана.
6. После “да” — реализуется фронт, запускаются проверки, делается review, выдаётся PR summary.

Важно: по нашим правилам оркестратор не должен править файлы до подтверждения плана.

### 3.4) Если спека “не готова”

Оркестратор вернёт список блокеров и вопросов к аналитику/бэку/дизайнеру.

Это ожидаемо и правильно: агент не должен “догадывать” backend contract.

Ускоряет работу, если вы сразу ответите на вопросы и обновите `docs/specs/GB-123.md`, после чего снова запустите:

```text
/work:orchestrate Продолжаем GB-123. Я обновил спеку, проверь ещё раз и дай план.
```

### 3.5) Если у вас GitBucket MCP

Если MCP уже подключён разработчиком, вместо файла можно дать ссылку/ID спеки:

```text
/work:orchestrate Реализовать GB-123. Спека в GitBucket: <ссылка>. Дизайн в Pixso: <ссылка>.
```

Если MCP не подключён, используйте вариант с `docs/specs/...` в репозитории.

## 4) Быстрые команды без полного цикла

Только валидировать спеку:

```text
/work:validate-spec Спека: docs/specs/GB-123.md
```

Только сделать план (без изменений):

```text
/work:plan Задача GB-123. Спека: docs/specs/GB-123.md. Дизайн: docs/design/GB-123.md.
```

Только review текущего diff:

```text
/work:review Проверь текущие изменения по задаче GB-123.
```

Сгенерировать PR summary:

```text
/work:pr-summary Сформируй PR summary по текущим изменениям.
```

## 5) Как использовать superpowers skills

Superpowers skills импортированы в extension и доступны Qwen.

Практически:

- Для новых/сложных фич: попросите использовать `brainstorming` перед реализацией.
- Для отладки: попросите использовать `systematic-debugging`.
- Для “не заявлять готовность без фактов”: используйте `verification-before-completion`.
- Для составления подробного плана: `writing-plans`.

Если вы хотите явно вызвать skill, прямо напишите в чате:

```text
Используй skill systematic-debugging и разберись, почему падают тесты.
```

или

```text
Используй skill writing-plans и составь подробный план реализации GB-123.
```

## 6) Рекомендации по дисциплине (чтобы работало стабильно)

- Держите `component-inventory` актуальным.
- Держите `local-verification` актуальным.
- Спеку и дизайн лучше сохранять в проект (`docs/specs/...`, `docs/design/...`).
- При изменениях больших проектных компонентов всегда фиксируйте в PR summary “какие страницы могут затронуться”.
