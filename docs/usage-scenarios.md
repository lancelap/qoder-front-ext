# Сценарии использования Qoder Frontend Workflow

Этот документ описывает, когда использовать полный workflow, а когда лучше идти короткими шагами.

## Базовый принцип

Если задача неочевидная или большая, сначала валидируйте входные данные и соберите план.

Если задача визуальная и её легко разрезать на изолированные этапы, не запускайте полный `/work:orchestrate` с самого начала.

## Сценарий 1. Есть сырая спека, реализацию начинать рано

Когда использовать:

- аналитик дал черновик;
- backend contract не до конца понятен;
- дизайн есть частично или ссылкой;
- высок риск, что агент начнёт додумывать поведение.

Что запускать:

1. `/work:validate-spec`
2. Ответить на блокеры.
3. Повторно `/work:validate-spec`
4. `/work:plan`

Чего не делать:

- не запускать сразу `/work:orchestrate`;
- не просить “сделай весь экран”.

## Сценарий 2. Есть Pixso-макет экрана, нужно сделать UI по шагам

Когда использовать:

- макет большой;
- есть риск, что оркестратор уйдёт в длинную full-cycle задачу;
- нужен управляемый diff.

Что запускать:

1. `/work:validate-spec`
2. `/work:plan` с требованием разбить на 5-7 итераций
3. По одному implementation prompt на этап

Используйте пакет:

- `docs/prompt-packs/pixso-screen-iteration/`

Порядок:

1. `00-validate-spec.md`
2. `01-plan.md`
3. `02-layout.md`
4. `03-filters.md`
5. `04-tabs-table.md`
6. `05-selection-states.md`
7. `06-polish-review.md`

## Сценарий 3. Нужно только проверить, готова ли задача к frontend-реализации

Когда использовать:

- до начала разработки;
- перед оценкой;
- когда нужно собрать вопросы к аналитику/дизайнеру.

Что запускать:

```text
/work:validate-spec Спека: <path>. Дизайн: <path-or-link>.
```

Ожидаемый результат:

- verdict `ready` или `blocked`;
- список дыр в контракте;
- список дыр в UI states;
- вопросы к смежным ролям.

## Сценарий 4. Нужен только план без правок

Когда использовать:

- перед созвоном;
- для оценки объёма;
- когда хочешь сам делать код, но нужен план от Qwen.

Что запускать:

```text
/work:plan Задача: <summary>. Спека: <path>. Дизайн: <path-or-link>.
```

Что просить дополнительно:

- likely files to change;
- reuse candidates;
- verification commands;
- risks and open questions.

## Сценарий 5. Реализация уже идёт, нужен только review текущего diff

Когда использовать:

- часть работы уже сделана руками;
- не нужен новый план;
- нужно проверить регрессии и соответствие спеке.

Что запускать:

```text
/work:review Проверь текущие изменения по задаче <task-id>.
```

Что ожидать:

- замечания по регрессиям;
- пробелы в состояниях;
- несоответствия спеке и дизайну;
- тестовые и verification gaps.

## Сценарий 6. Нужно только собрать PR summary

Когда использовать:

- работа уже завершена;
- не нужен orchestration;
- нужен текст для PR или handoff.

Что запускать:

```text
/work:pr-summary Сформируй PR summary по текущим изменениям.
```

## Сценарий 7. В проекте ещё нет контекста для Qwen

Когда использовать:

- новый frontend-репозиторий;
- нет `QWEN.md`;
- нет `.qwen/skills/component-inventory`;
- нет `.qwen/skills/local-verification`.

Что запускать:

```text
/work:init-project
```

Потом обязательно заполнить:

- `QWEN.md`
- `.qwen/skills/project-architecture/SKILL.md`
- `.qwen/skills/component-inventory/SKILL.md`
- `.qwen/skills/local-verification/SKILL.md`

## Сценарий 8. Оркестратор начинает делать слишком много

Симптомы:

- слишком длинный ответ;
- пытается сразу пройти validate -> plan -> implement -> test -> review;
- меняет больше файлов, чем ожидалось;
- строит лишние абстракции.

Что делать:

1. Остановить полный orchestration.
2. Перейти на `/work:validate-spec` и `/work:plan`.
3. Разбить задачу на короткие implementation prompts.
4. На каждом этапе явно писать `Out of scope`.

Рабочее правило:

- один prompt должен менять один логический слой: layout, filters, table, selection, states или polish.

## Сценарий 9. Нужно работать по Pixso, но без уверенности в картинках

Когда использовать:

- есть Pixso MCP, но хочется более предсказуемый workflow;
- есть только экспорт, DSL или текстовое описание;
- важнее контрактная реализация, чем “угадай по картинке”.

Что делать:

1. Сначала собрать текстовый DSL макета.
2. Проверить, что в DSL есть:
   - структура экрана;
   - поля формы;
   - таблица/колонки;
   - actions;
   - states;
   - sample data.
3. Передавать Qwen DSL + Pixso link + короткий scope этапа.

Почему это работает:

- меньше пространства для галлюцинаций;
- проще ревьюить;
- проще повторно запускать отдельный этап.

## Сценарий 10. Реализация почти верная, но есть точечные расхождения

Когда использовать:

- экран уже собран;
- не совпадает mapping полей;
- отдельный элемент ведёт себя не так;
- есть визуальные расхождения с макетом.

Что не делать:

- не перезапускать весь `/work:orchestrate`;
- не просить “исправь экран целиком”;
- не смешивать mapping, behavior и visual issues в один vague prompt.

Что делать:

1. Зафиксировать `Current result`.
2. Зафиксировать `Expected result`.
3. Явно перечислить `Out of scope`.
4. Выбрать correction prompt по типу mismatch.

Шаблоны:

- `docs/prompt-packs/pixso-screen-iteration/07-fix-mapping.md`
- `docs/prompt-packs/pixso-screen-iteration/08-fix-behavior.md`
- `docs/prompt-packs/pixso-screen-iteration/09-fix-visual-alignment.md`

Рабочее правило:

- один correction prompt исправляет один класс проблемы.

## Практические правила

- Не используйте `/work:orchestrate` как default entrypoint для любой UI-задачи.
- Для новых экранов сначала лучше `validate-spec`, потом `plan`.
- Для сложных экранов просите итерационный план, а не единый implementation plan.
- Для каждого implementation prompt явно пишите `Scope`, `Out of scope`, `Acceptance`, `Constraints`.
- Если задача визуальная, лучше 5 коротких промптов, чем один “сделай всё”.
